# 資料控管概觀

Adobe Experience Platform的核心功能之一，是將來自多個企業系統的資料彙集在一起，讓行銷人員更能識別、瞭解客戶並與客戶互動。 此資料可能受貴組織或法律法規所定義的使用限制所約束。 因此，請務必確保您在下列範圍內的資料作業： [!DNL Platform] 符合資料使用原則。

Adobe Experience Platform資料控管可讓您管理客戶資料，並確保遵守適用於資料使用的法規、限制和原則。 它在以下方面發揮著關鍵作用 [!DNL Experience Platform] 在不同的層級，包括編目、資料譜系、資料使用標籤、資料使用原則，以及控制行銷動作的資料使用。

>[!NOTE]
>
>在Experience Platform中，資料控管只關心資料的使用或啟用方式，無論使用者執行的是什麼。 如需如何控制組織內特定Platform使用者對特定資料欄位之存取許可權的詳細資訊，請參閱以下檔案： [基於屬性的存取控制](../access-control/abac/overview.md) 而非。

## 資料控管角色

作為一個概念，資料控管既不是自動的，也不是在真空中進行。 起初只是一個人的角色，通常被認為是資料管理員，但隨著資料控管生態系統的擴展，該角色已大幅成長。 今天，資料控管需要持續管理和監控才能成功，並且仰賴資料管理員擁有可正確標籤資料、建立使用原則以及強制遵守這些原則的工具。

雖然資料控管應該是組織中每個人的責任，但以下是資料控管週期中的一些基本角色：

![資料控管角色](./images/overview/roles.png)

### 資料管理員

資料管理員是資料控管的核心。 此角色負責解譯法規、合約限制和原則，並將其直接套用至資料。 透過他們對這些法規、限制和政策的瞭解所知，資料管理員的角色包括：

* 檢閱資料、資料集和資料範例，以套用和管理中繼資料使用標籤。
* 建立資料原則並將其套用至資料集和欄位。
* 將資料原則通訊給組織。

### 行銷人員

行銷人員是資料控管的終點。 他們從資料管理員、科學家和工程師建立的資料控管基礎架構要求資料。 行銷人員在行銷傘狀結構下包含許多不同的專長，其中包括：

* 行銷分析人員需要資料，以便瞭解個人和群組（也稱為區段）客戶。
* 行銷專家與體驗設計人員會使用資料設計新的客戶體驗。


## 資料控管框架

資料控管架構可簡化及簡化分類資料和建立資料使用原則的流程。 套用資料標籤並建立資料使用原則後，即可評估行銷動作，以確保正確使用資料。

資料控管框架有三個關鍵元素：標籤、原則和執行。

1. **標籤：** 將反映隱私權相關考量事項和合約條件的資料分類，以符合法規和組織政策。
1. **原則：** 說明允許或不允許對特定資料執行哪些行銷動作。
1. **強制執行：** 使用原則架構，在不同資料存取模式間提供建議並強制執行原則。

## 資料使用標籤

資料控管可讓資料管理員在結構欄位層級和資料集層級套用使用標籤，以便根據套用的原則型別來分類資料。

資料控管架構包含預先定義的資料使用標籤，可透過三種方式將資料分類：

![資料使用標籤類別](./images/overview/label-categories.png)

* **合約「C」資料標籤：** 為具有合約義務或與客戶資料治理原則相關的資料加上標籤並加以分類。
* **身分「I」資料標籤：** 為可識別或聯絡特定人員的資料加上標籤並加以分類。
* **敏感的「S」資料標籤：** 標示和分類與敏感資料（例如地理資料）相關的資料。

>[!NOTE]
>
>請參閱指南： [支援的資料使用標籤](labels/reference.md) 以取得可用標籤的完整清單，以及每個標籤型別的定義。

標籤可隨時套用，讓您靈活選擇控管資料的方式。 最佳實務建議在資料內嵌至後立即加上標籤 [!DNL Experience Platform]，或在資料在中成為可用資料時 [!DNL Platform].

請參閱以下文章的概觀： [資料使用標籤](./labels/overview.md) 以取得詳細資訊。

## 資料使用原則 {#data-usage-policies}

為了使資料使用標籤有效地支援資料合規性，必須實施資料使用原則。 資料使用原則是描述允許或限制您對內資料執行何種行銷動作的規則 [!DNL Experience Platform].

行銷動作的範例可能是想要將資料集匯出至協力廠商服務。 如果制定原則宣告個人識別資訊(PII)無法匯出，且「I」標籤（身分資料）已套用至資料集， [!DNL Policy Service] 防止任何會將此資料集匯出至協力廠商目的地的動作。 如果其中一個動作嘗試發生，原則服務會傳送訊息告訴您資料使用原則已違反。

有兩種可用的原則：

* **[!UICONTROL 資料治理原則]**：根據執行的行銷動作和有問題資料攜帶的資料使用標籤，限制資料啟用。
* **[!UICONTROL 同意原則]**：篩選可以啟用的設定檔 [目的地](../destinations/home.md) 根據客戶的同意或偏好設定。

套用資料使用標籤後，資料管理員可以使用 [!DNL Policy Service] API或 [!DNL Experience Platform] 使用者介面。 如需資料使用原則和行銷動作的詳細資訊，請參閱 [原則概觀](./policies/overview.md).

>[!IMPORTANT]
>
>所有資料使用原則(包括Adobe提供的核心原則)預設為停用。 為了將個別原則考慮為強制執行，您必須手動啟用該原則。

### 資料控管術語 {#data-governance-terminology}

下表概述與資料控管和資料控管框架相關的關鍵術語。

| 詞語 | 定義 |
|---|---|
| **合約標籤** | 合約「C」標籤可用來對具有合約義務或與貴組織的資料治理原則相關的資料進行分類。 |
| **跨網站資料** | 跨網站資料是來自多個網站的資料組合，包括站上資料和站外資料的組合，或來自多個站外來源的資料組合。 |
| **資料控管** | 資料控管涵蓋了用於確保資料在資料使用方面符合法規和公司政策的策略和技術。 |
| **資料管理員** | 資料管理員是負責管理、監督及執行組織資料資產的人員。 資料管理員也可確保資料控管政策受到保護，並維持其符合政府法規與組織政策。 |
| **資料使用標籤** | 資料使用標籤讓使用者能夠分類反映隱私權相關考量事項，以及遵循法規和公司政策的合約條件的資料。 |
| **資料集標籤** | 標籤可以新增到結構描述。 資料集內的所有欄位都會繼承結構描述的標籤。 |
| **欄位標籤** | 欄位標籤是繼承自結構描述或直接套用至欄位的資料治理標籤。 套用至欄位的資料治理標籤不會繼承到結構描述層級。 |
| **地理圍欄** | 地理圍欄是虛擬的地理邊界，由GPS或RFID技術定義，可讓軟體在行動裝置進入或離開特定區域時觸發回應。 |
| **身分標籤** | 身分「I」標籤可用來對可識別或聯絡特定人員的資料進行分類。 |
| **興趣型目標定位** | 如果符合下列三個條件，就會發生以興趣為基礎的鎖定目標（也稱為個人化）：在網站上收集的資料用於推斷使用者的興趣，而在其他內容中使用，例如在另一個網站或應用程式（離站），並用於根據這些推斷選擇提供哪些內容或廣告。 |
| **行銷動作** | 在資料控管架構的情境下，行銷動作是指 [!DNL Experience Platform] 資料取用者需要檢查資料使用原則是否違規 |
| **政策** | 在資料控管架構中，原則是描述允許或不允許對特定資料執行何種行銷動作的規則。 |
| **結構描述標籤** | 管理架構層級的資料控管、同意和存取控制標籤。 這會將標籤推廣至使用該結構描述的每個資料集。 |
| **敏感標籤** | 敏感「S」標籤可用來對您及貴組織認為敏感的資料進行分類。 |
