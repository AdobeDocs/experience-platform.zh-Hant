# 標籤的使用者權限

>[!NOTE]
>
>Adobe Experience Platform Launch在Adobe Experience Platform中已重新命名為一套資料收集技術。 因此，所有產品文件中出現了幾項術語變更。 如需術語變更的彙整參考資料，請參閱以下[文件](../../term-updates.md)。

Adobe Experience Platform中標籤的使用者權限會透過Adobe Admin Console指派給使用者。 不同的權限集會以產品設定檔的形式個別設定，而非指派給個別使用者。 接著，會將使用者指派給這些產品設定檔，以授予他們已設定的權限。

本指南概述標籤可用的不同類型權限、其所授予的存取權，以及不同業務使用案例的一些基本實作策略。

>[!NOTE]
>
>有關如何使用Admin Console配置用戶權限的步驟，請參閱 [管理資料收集的權限](../../../collection/permissions.md).

## 權限類型

在產品設定檔中，標籤的權限分為四個類別：

1. 平台
1. 屬性
1. 屬性權利
1. 公司權利

### 平台

每個標籤屬性都有一個平台。 目前有兩個平台可供標籤使用：網路和行動。 您可以善用此權限類型，限制或授予特定屬性類型的存取權。當管理您行動應用程式的團隊與管理您網站的團隊不同時，這個功能會相當實用。

### 屬性

依預設，產品設定檔會授予您公司中目前和未來所有屬性的存取權。 使用此權限類型，您可以依名稱限制或授予特定現有屬性的存取權。

### 屬性權利 {#property-rights}

您在資料收集UI中建立的任何屬性都可在Admin Console中使用，讓您可將屬性以相同產品設定檔中的特定屬性權限分組。

例如，如果指定的產品設定檔沒有屬性A1的存取權，屬於該設定檔的使用者將看不到或修改屬性A1內的任何設定。

如果使用者屬於某個具有屬性A1存取權的設定檔，則他們可以在屬性A1內執行的動作，會由該設定檔所授予的權限決定。 如果使用者擁有屬性A1的權限，但沒有指派的權限，則他們擁有該屬性的唯讀存取權。

下表概述可用的屬性權利及其授予的存取權：

| 屬性權利 | 說明 |
| --- | --- |
| **開發** | 這可讓您執行下列動作：<ul><li>建立規則和資料元素</li><li>建立程式庫並在現有開發環境中建置</li><li>提交程式庫以進行核准</li></ul>資料收集UI中的大部分日常工作都需要此權限。 |
| **核准** | 這可讓您取得已提交的程式庫，並建置至測試環境。 完成測試之後，您也可以核准程式庫以進行發佈。 |
| **發佈** | 這可讓您將已核准的程式庫發佈至生產環境。 |
| **管理擴充功能** | 這可讓您執行下列動作： <ul><li>將新擴充功能安裝至屬性</li><li>修改已安裝擴充功能的設定</li><li>刪除擴充功能</li></ul>請參閱的擴充功能概觀檔案 [擴充功能的詳細資訊](../managing-resources/extensions/overview.md). 視您的組織而定，此角色通常屬於 IT 或行銷部門。 |
| **管理環境** | 這可讓您建立和修改環境。 請參閱 [環境檔案](../publishing/environments.md) 以取得更多資訊。 此角色通常屬於 IT 群組。 |

### 公司權利

公司權利適用於橫跨多個屬性的權限。下表概述了這些問題：

| 公司權利 | 說明 |
| --- | --- |
| **管理屬性** | 這可讓您執行下列動作：<ul><li>建立新屬性</li><li>在屬性層級修改中繼資料和設定</li><li>刪除屬性</li></ul>管理員通常會執行此角色。請參閱 [屬性檔案](companies-and-properties.md) 以取得更多資訊。 |
| **開發擴充功能** | 授予建立和修改公司擁有的擴充功能套件的能力，包括私人發行和公開發行請求。 |
| **管理應用程式設定** | 只有在您擁有Adobe Journey Optimizer的授權，或其他授予行動應用程式內及推送訊息存取權的解決方案，才能使用此功能。  這可讓您管理Experience Cloud所知的應用程式，以及與Firebase雲端訊息服務和Apple推播通知服務通訊所需的必要推播憑證。 |

## 總計使用者權限

個別使用者的總權限取決於其不同產品設定檔中的總成員資格。 如果使用者屬於多個產品設定檔，來自每個設定檔的權限會相加在一起，而非相乘。

例如，產品設定檔A授予您屬性1的開發權利。 產品設定檔B授予您屬性2的發佈權限。 在此情況下，您可以在屬性1中開發，在屬性2中發佈，但無法在屬性1中發佈，或在屬性2中開發，因為您尚未被授予執行此操作的明確權限。

## 權利案例

建立新產品設定檔時，不同的公司有不同的需求。 這些需求會依公司規模、組織結構、網站數量、參與管理標籤的人員數量等而有所不同。

在您思考如何建立產品設定檔及將使用者新增至產品設定檔時，以下是一些常見案例和建議的起點。

### 單人負責

如果您經營一家由一人負責所有事務的小型公司，請授予此使用者所有屬性的權限，並將上述所有權限指派給他們。

### 責任分散

請考量組織中許多人參與標籤的情況。 您有一組會建立規則和資料元素的人員（例如外部顧問），但您不希望他們能存取生產環境。 在此情況下，您要確定除了IT團隊外，沒有人部署至生產環境。

若要完成此作業：

1. 為顧問建立帳戶，並僅授予顧問「開發」權利。
1. 顧問可在您設定的限制內進行建置及測試。
1. 如果顧問想要新的擴充功能或已準備好上線，貴組織的代表（具有適當權限）會執行這些動作。

### 企業

企業公司可能有分散在不同地理位置的多個網站，以及負責各個地理位置的不同團隊。在這些團隊內，由不同的個人負責開發及發佈。

這類似於上述的「責任分散」，不過是依地理位置區域而加以編排。例如，您可以為北美地區建立「開發」設定檔和「發佈」設定檔，並為歐洲建立個別的「開發」和「發佈」群組。

## 範例角色

下表提供您在組織中可能擁有的角色類型以及您應指派哪些權限的一些範例：

| 角色 | 說明 | 屬性 | 屬性權利 | 公司權利 |
| --- | --- | --- | --- | --- |
| 主管 | 想要查看系統中發生的情況，但無法進行任何變更。 | 自動納入 | (None) | （無） |
| 行銷人員 | 可以安裝擴充功能並為現有屬性設定新標籤，但無法發佈至測試環境或生產環境。 | 自動納入 | <ul><li>開發</li><li>管理擴充功能</li></ul> | <ul><li>管理屬性</li></ul> |
| 行動應用程式開發人員 | 負責在原生行動應用程式內實作Adobe和第三方解決方案。 | 自動納入 | <ul><li>開發</li><li>管理擴充功能</li></ul> | <li>管理屬性</li><li>管理應用程式設定</li> |
| IT 團隊 | 實際上不會修改任何標籤，但可完全控制測試環境和生產環境，以及其中的內容。 | 自動納入 | （無） | <ul><li>核准</li><li>發佈</li><li>管理環境</li></ul> |
| 擴充功能開發人員 | 開發擴充功能並可提交以進行核准，但無法發佈或新增至現有屬性。 | 自動納入 | <ul><li>開發</li></ul> | <ul><li>管理屬性</li><li>開發擴充功能</li></ul> |
| 超級用戶 | 做所有事。 | 自動納入 | <ul><li>開發</li><li>核准</li><li>發佈</li><li>管理擴充功能</li><li>管理環境</li></ul> | <ul><li>管理屬性</li></ul> |
