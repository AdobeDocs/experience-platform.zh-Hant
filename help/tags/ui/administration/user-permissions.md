# 標籤的用戶權限

>[!NOTE]
>
>Adobe Experience Platform Launch已被改名為Adobe Experience Platform的一套資料收集技術。 因此，所有產品文件中出現了幾項術語變更。 如需術語變更的彙整參考資料，請參閱以下[文件](../../term-updates.md)。

Adobe Experience Platform中標籤的用戶權限通過Adobe Admin Console分配給用戶。 不同的權限集不是分配給單個用戶，而是單獨配置為產品配置檔案。 然後，用戶將被分配到這些產品配置檔案，以便被授予為其配置的權限。

本指南概述了可用於標籤的不同類型的權限、它們授予的訪問權限以及針對不同業務使用情形的一些基本實施策略。

>[!NOTE]
>
>有關如何為使用Admin Console的用戶配置權限的步驟，請參閱上面的教程 [管理資料收集權限](../../../collection/permissions.md)。

## 權限類型

在產品配置檔案中，標籤的權限分為四類：

1. 平台
1. 屬性
1. 屬性權利
1. 公司權利

### 平台

每個標籤屬性都具有平台。 當前有兩個平台可用於標籤：Web和Mobile。 您可以善用此權限類型，限制或授予特定屬性類型的存取權。當管理您的移動應用的團隊與管理您網站的團隊不同時，此功能將非常有用。

### 屬性

預設情況下，產品配置檔案授予您對公司內當前和將來的所有屬性的訪問權限。 使用此權限類型，可以按名稱限制或授予對特定現有屬性的訪問權限。

### 屬性權利 {#property-rights}

在「資料收集UI」中建立的任何屬性都將在Admin Console中可用，從而允許您將屬性與同一產品配置檔案中的特定屬性權限分組。

例如，如果給定的產品配置檔案沒有訪問屬性A1的權限，則屬於該配置檔案的用戶將無法查看或修改屬性A1中的任何設定。

如果用戶屬於確實有權訪問屬性A1的配置檔案，則他們可以在屬性A1中執行的操作由他們從此配置檔案中授予的權限決定。 如果用戶對屬性A1具有權限但沒有分配權限，則他們對該屬性具有只讀訪問權限。

下表概述了可用的屬性權限及其授予訪問權限的功能：

| 產權 | 說明 |
| --- | --- |
| **開發** | 這允許您執行以下操作：<ul><li>建立規則和資料元素</li><li>建立庫並在現有開發環境中構建庫</li><li>提交庫以供審批</li></ul>資料收集UI中的大多數日常任務都需要此權限。 |
| **核准** | 這允許您將已提交的庫並構建到暫存環境。 完成測試之後，您也可以核准程式庫以進行發佈。 |
| **發佈** | 這允許您將批准的庫發佈到生產環境。 |
| **管理擴展** | 這允許您執行以下操作： <ul><li>為屬性安裝新擴展</li><li>修改已安裝的擴展的配置</li><li>刪除擴展</li></ul>請參閱擴展概述文檔 [有關擴展的詳細資訊](../managing-resources/extensions/overview.md)。 視您的組織而定，此角色通常屬於 IT 或行銷部門。 |
| **管理環境** | 這允許您建立和修改環境。 查看 [環境文檔](../publishing/environments.md) 的子菜單。 此角色通常屬於 IT 群組。 |

### 公司權利

公司權利適用於橫跨多個屬性的權限。下表概述了這些內容：

| 公司權 | 說明 |
| --- | --- |
| **管理屬性** | 這允許您執行以下操作：<ul><li>建立新屬性</li><li>修改屬性級別的元資料和設定</li><li>刪除屬性</li></ul>管理員通常會執行此角色。查看 [屬性文檔](companies-and-properties.md) 的子菜單。 |
| **開發擴展** | 授予建立和修改公司擁有的擴展包的能力，包括私有版本和公開版本請求。 |
| **管理應用程式配置** | 僅當您擁有Adobe Journey Optimizer的許可證或授予訪問移動應用內和推送消息的權限的其他解決方案時，才可使用此功能。  這允許您管理Experience Cloud知道的應用以及與Firebase雲消息服務和Apple推送通知服務通信所需的推送憑據。 |

## 總計使用者權限

單個用戶的總權限由其在不同產品配置檔案中的總成員資格確定。 如果用戶屬於多個產品配置檔案，則每個配置檔案的權限將相加而不是相乘。

例如，產品配置檔案A授予您屬性1的開發權。 產品配置檔案B授予您屬性2的發佈權限。 在這種情況下，您可以在屬性1中開發，在屬性2中發佈，但不能在屬性1中發佈或在屬性2中開發，因為您尚未被授予執行此操作的明確權限。

## 權利案例

不同公司在建立新產品配置檔案時有不同的需求。 這些需求因公司規模、組織結構、站點數量、管理標籤的人員數量等而異。

下面是一些常見情形，以及您考慮建立產品配置檔案和向其添加用戶時建議的起點。

### 單人負責

如果您運行的小公司由一名負責所有事務的人員負責，請為所有屬性授予此用戶權限，並為其分配上面列出的所有權限。

### 責任分散

請考慮組織中許多人參與標籤的情況。 您有一組人員（如外部顧問），他們建立規則和資料元素，但您不希望他們具有對生產環境的訪問權限。 在這種情況下，您需要確保除IT團隊外沒有人部署到生產部門。

為此，請執行以下操作：

1. 為顧問建立帳戶並僅授予他們「開發」權限。
1. 顧問可在您設定的限制內進行建置及測試。
1. 如果顧問想要新的副檔名或準備投入使用，則您的組織（具有相應權限）的代表將執行這些操作。

### 企業

企業公司可能有分散在不同地理位置的多個網站，以及負責各個地理位置的不同團隊。在這些團隊內，由不同的個人負責開發及發佈。

這類似於上述的「責任分散」，不過是依地理位置區域而加以編排。例如，您可以為北美建立「開發」配置檔案和「發佈」配置檔案，並為歐洲建立單獨的「開發」和「發佈」組。

## 示例角色

下表提供了您在組織中可能具有的角色類型以及您應該分配哪些權限的一些示例：

| 角色 | 說明 | 屬性 | 屬性權利 | 公司權利 |
| --- | --- | --- | --- | --- |
| 主管 | 希望瞭解系統中正在發生的情況，但不能進行任何更改。 | 自動納入 | (None) | （無） |
| 行銷人員 | 可以為現有屬性安裝擴展並設定新標籤，但無法發佈到暫存或生產環境。 | 自動納入 | <ul><li>開發</li><li>管理擴展</li></ul> | <ul><li>管理屬性</li></ul> |
| 行動應用程式開發人員 | 負責在本機移動應用中實施Adobe和第三方解決方案。 | 自動納入 | <ul><li>開發</li><li>管理擴展</li></ul> | <li>管理屬性</li><li>管理應用程式配置</li> |
| IT 團隊 | 實際上不會修改任何標籤，但是它們完全控制了轉移和生產環境以及其中的內容。 | 自動納入 | （無） | <ul><li>核准</li><li>發佈</li><li>管理環境</li></ul> |
| 擴充功能開發人員 | 開發擴展並可以提交以供審批，但無法發佈或將其添加到現有屬性。 | 自動納入 | <ul><li>開發</li></ul> | <ul><li>管理屬性</li><li>開發擴展</li></ul> |
| 超級用戶 | 做了一切。 | 自動納入 | <ul><li>開發</li><li>核准</li><li>發佈</li><li>管理擴展</li><li>管理環境</li></ul> | <ul><li>管理屬性</li></ul> |
