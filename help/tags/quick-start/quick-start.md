# 快速入門手冊

>[!NOTE]
>
>Adobe Experience Platform Launch已被改名為Adobe Experience Platform的一套資料收集技術。 因此，所有產品文件中出現了幾項術語變更。 如需術語變更的彙整參考資料，請參閱以下[文件](../term-updates.md)。

標籤是Adobe Experience Platform的下一代標籤管理技術。 它從頭開始構建，以支援一個開放和可持續的生態系統，在這個生態系統中，任何人都可以構建自己的整合，讓Adobe客戶可以將整合部署到其站點。 這是 API 優先的應用程式，所以您可以透過 UI 完成的任何工作，也可以透過 API 以程式設計方式完成。

基本標籤工作流：

1. 設定群組和使用者。
2. 登入。
3. 建立屬性。
4. 安裝擴充功能。
5. 建立資料元素和規則。
6. 在您的開發環境中測試。
7. 提升至生產環境。

## 1. 設定群組和使用者

標籤與您的Adobe ID完全整合。 用戶權限通過與Adobe產品和解決方案的Admin Console進行管理 [!DNL Creative Cloud]。 [!DNL Document Cloud]和Experience Cloud。

標籤具有基於權限的用戶管理系統。 這意味著必須明確授予個人權利。 這些權限會指派給群組，然後使用者加入適當的群組以取得存取權限。即使您的組織有權訪問資料收集，在管理員明確授予其某些權限之前，單個用戶也無法執行任何操作。

有關如何建立組和添加標籤用戶的詳細說明，請參閱 [資料收集權限指南](../../collection/permissions.md)。

## 2. 登入

在將標籤權限添加到您的Adobe ID後，您需要登錄到Experience PlatformUI或資料收集UI。 通過直接導航到 [Experience Cloud登錄螢幕](https://experience.adobe.com/)，選擇 **[!UICONTROL 資料收集]** 或 **[!UICONTROL Experience Platform]**。

>[!NOTE]
>
>如果您有一個帳戶，並且有權訪問多個組織，則可以通過在螢幕頂部的控制欄中選擇組織名稱並從下拉清單中選擇其他組織來更改組織。

## 3. 建立屬性

登錄UI後，首先要建立屬性。 屬性基本上是個容器，當您將標籤部署至網站時，在其中裝入擴充功能、規則、資料元素和程式庫。許多人會針對要部署相同標籤集的每個網站 (或具有密切關聯的網站群組) 建立屬性。

如需有關建立屬性的詳細資訊，請參閱[建立屬性](../ui/administration/companies-and-properties.md)。

## 4. 安裝擴充功能

擴展是由Adobe或Adobe合作夥伴構建的整合，它為您可以部署到站點的標籤添加了新的、無盡的選項。 如果您將標籤視為作業系統，則擴展是您安裝的應用，用於執行需要它執行的特定操作。

所有全新屬性都會隨[核心擴充功能](../extensions/client/core/overview.md)一併安裝。行動屬性會隨其他擴充功能提供。核心擴展通過Adobe構建，為資料層和規則的事件類型提供一組強健的預設資料元素類型。 大部分您希望執行的動作 (取得 ECID、傳送 [!DNL Adobe Analytics] 信標、載入 [!DNL Target] 全域 mbox 等) 會來自您從目錄安裝的擴充功能。

使平台中的標籤真正獨一無二的是，這些擴展可以由任何人構建。 您是否需要將 Facebook 再行銷像素放置在您的網站上？試試看 Facebook 建立的擴充功能。您是否要在 Twitter 或 LinkedIn 提供相同體驗？請使用這些擴充功能。您需要執行調查嗎？您需要的是 Question Pro 或 Foresee。您是否需要管理您的最終用戶的隱私和同意，以幫助 [!DNL GDPR]? 請深入了解 Evidon 和 Trust Arc。是否希望詳細瞭解站點上單個用戶的行為？ 也許您可以了解一下 Clicktale。有關詳細資訊，請參閱上的 [添加新擴展](../ui/managing-resources/extensions/overview.md#add-a-new-extension)。

## 5. 建立資料元素和規則

**資料元素**&#x200B;是指標，指向您希望收集並傳送至頁面上不同位置的資訊：

* JSON 中定義的資料層
* DOM 元素
* Cookie
* 工作階段與本機儲存
* 幾乎其他所有資訊

定義資料元素後，可以在整個UI中的任何位置使用該元素進行任何擴展。 請參閱 [資料元素](../ui/managing-resources/data-elements.md) 的子菜單。

**規則**&#x200B;位於實作的邏輯核心，並控制網站上所有標籤的內容、出現時間與位置以及運作方式。定義事件、設定條件和例外，然後定義動作和順序。最後，發佈變更以查看結果。如需詳細資訊，請參閱[規則](../ui/managing-resources/rules.md)。

## 6. 在您的開發環境中測試

### 程式庫與組建

標籤生成從不自動發佈。 您所做的每一組變更都會封裝到[程式庫](../ui/publishing/libraries.md)中。您建立的每個程式庫都會自動繼承任何上游的內容 (已發佈、已核准或已提交) 作為基準，因此您只需要定義您要做的變更。此程式庫可作為[組建](../ui/publishing/builds.md)的藍圖。組建是指部署和使用的實際 JavaScript 檔案集。

瞭解網頁、托管位置和標籤之間的關係非常重要。

1. 主機伺服器提供發佈生成的位置。 生成本身包含庫所需的JavaScript檔案。

   每個環境都與主機有關係，並且主機提供一個端點，指示要在何處傳送生成。 該主機只能屬於一個屬性，儘管一個屬性可以包含多個主機。

2. 表單中提供了嵌入代碼  `<script>` 進入到 `<head>` 的下界HTML。

   當您建立環境並連接主機時，該環境將自動生成唯一的嵌入代碼，允許您將其分配的內部版本整合到您的站點中。 的 `<script>` 代碼用於在運行時部署庫生成。

3. 當用戶瀏覽您的站點時，嵌入代碼 `<script>` 標籤從主機伺服器檢索生成並在瀏覽器中執行您定義的操作。

### 主機

主機是標籤屬性與宿主位置之間的連接。 標籤當前支援通過Adobe托管 [!DNL Akamai] 主機或通過SFTP主機自主承載。 每當生成生成時，標籤都會連接到由主機定義的伺服器並傳遞生成。

如果您是自主托管，則標籤生成可以通過SFTP直接推送到您的伺服器，或者您可以將其推送到 [!DNL Akamai] 並使用您環境的「存檔」選項下載。

如需詳細資訊，請參閱[主機](../ui/publishing/hosts/hosts-overview.md)。

### 環境

每個程式庫都會在環境中建立。環境會定義您希望在組建發佈時呈現的外觀。您可以指定在網站上傳遞：

* **主機：** 每個環境都需要一個主機，該主機確定將在此環境中建立的任何生成推送到的終結點。
* **存檔：** 預設設定是將生成部署為微型.js檔案。 如果使用自定義代碼，則可能有多個檔案相互引用。 這些檔案可以合併為單個zip檔案並加密。

儲存您的環境後，環境會產生內嵌程式碼，您可以複製程式碼並貼到您的網站上。請注意，在您建立庫並生成內部版本之前，嵌入代碼不會工作。 如需詳細資訊，請參閱[環境](../ui/publishing/environments.md)。

### 將組建發佈至開發環境

發佈過程在以下步驟中介紹。

1. 建立主機。
1. 使用您建立的主機建立開發環境。
1. 將內嵌程式碼從開發環境部署到開發測試網站。
1. 建立程式庫並將其指派至您建立的開發環境。
1. 建立程式庫。

## 7. 提升至生產環境

在開發環境中測試了您的構建後，請確保建立您的階段和生產環境並將嵌入的代碼放在必要的位置。 您可以為此目的重新使用現有主機。

將程式庫完全提升至生產環境，通常需要協調具有不同權限的使用者。

* 開發人員 (具有「開發」權限的人) 會提交程式庫，這樣會將程式庫移至「已提交」狀態。
* 核准者 (具有「核准」權限的人) 可以建立程式庫至預備環境，並在測試後核准。這樣會將程式庫移至「已核准」狀態。一次只能提交和核准一個程式庫。
* 發佈者 (具有「發佈」權限的人) 可以建立程式庫至生產環境。

您可以將所有這些權限指派給同一人員。

如需發佈程序期間不同的可用狀態和選項的詳細資訊，請參閱[核准工作流程](../ui/publishing/publishing-flow.md)。
